人们从 ChatGPT 中激发出的幽默与创造力——“像对一只狗一样给我解释 HTML”、“为 Chess 2 写发布说明”——都源于同一机制。从模型的视角来看，每一次交互都只是一个序列补全问题。你提供一段序列的开头，模型便根据它所学到的语言行为模式来续写这段序列 [00:01:06]。

这一切的背后，是一种名为 Transformer 的特定神经网络架构。Transformer 在 2017 年的论文 Attention Is All You Need 中被首次提出，最初用于机器翻译。其作者很可能没有预料到，仅需少量修改，这一架构便会在随后五年里被复制粘贴到几乎所有主流 AI 应用中，包括 ChatGPT 的核心 [00:02:07]。GPT 本身代表 Generatively Pre-trained Transformer（生成式预训练 Transformer）。

在单个教程中完整复现 ChatGPT 是不可能的：它是在海量互联网数据上训练，并经过多轮预训练与微调阶段打磨而成。相反，我们将专注于构建一个规模小得多、但教育意义丰富的系统——一个基于 tiny Shakespeare 数据集训练的字符级语言模型。该文件约一兆字节，包含了莎士比亚的全部作品。我们的 Transformer 将学习建模字符之间的接续关系：给定一段前置文本，预测下一个字符 [00:03:40]。训练完成后，模型可以生成无限的伪莎士比亚文本，一次一个字符地产出诸如 “verily my Lord the sites have left the again the king coming with my curses...” 这样的段落 [00:04:40]。

我们将从零开始编写的实现基于 NanoGPT，这是一个极简的代码仓库，大约只有两个文件、每个文件 300 行代码——一个定义 GPT 模型，一个处理训练。尽管 NanoGPT 可以扩展以在更大的数据集（如 OpenWebText）上复现 GPT-2（1.24 亿参数）这样的模型，但我们的目标是从一个空文件开始，逐步构建 Transformer，在 tiny Shakespeare 上训练，并生成文本。唯一的先决条件是熟练掌握 Python、具备基础的微积分与统计学知识，以及熟悉基本的神经网络概念 [00:05:43]。

加载数据集并构建词表

tiny Shakespeare 数据集是一个约 1 MB 的文本文件，从公开 URL 下载。读入后可以发现它包含大约一百万个字符。打印前 1,000 个字符，可以确认其为预期的莎士比亚式散文。

为了确定模型使用的字母表，我们将完整文本传入 Python 的 set 构造函数以收集每一个唯一字符，再将结果转换为 list 并进行 sort。这样就得到了一个包含 65 个字符的词表：空格、各种标点符号和特殊字符、大写字母以及小写字母。这个排序后列表的大小——65——就是我们的词表大小(vocabulary size)，即我们的序列可以包含或生成的所有可能元素的集合。[00:08:17]

字符级分词：编码器与解码器

分词(tokenization)是指根据固定的词表将原始文本转换为整数序列。在本讲座中，我们使用最简单的方案：字符级语言模型，其中每个独立字符映射到一个唯一整数。我们同时构建编码器（string → list of integers）和解码器（list of integers → string），使得转换是无损且可逆的。[00:09:21]

stoi = { ch:i for i,ch in enumerate(chars) }
itos = { i:ch for i,ch in enumerate(chars) }
encode = lambda s: [stoi[c] for c in s]
decode = lambda l: ''.join([itos[i] for i in l])

print(encode("hi there"))
print(decode(encode("hi there")))

这只是众多可能的分词方案之一。在实践中，模型通常使用子词(subword)分词器。例如，Google 的 SentencePiece 在子词级别而非单个字符级别对文本进行编码，而 OpenAI 的 tiktoken 库实现了一个字节对编码(byte-pair encoding, BPE)分词器，其词表大小约为 50,000 个 token。使用如此大的词表，同样的字符串 "hi there" 可能只压缩为三个整数，但这些整数的取值范围要大得多（最高约 50,256）。其中的权衡很明显：更小的词表会产生更长的序列，而更大的词表则会产生更短的序列。我们这里坚持使用字符级分词，因为极小的词表和简单的编解码逻辑能让代码保持清晰透明。[00:10:54]

用于语言建模的批量分块采样

Transformers 在整数序列上进行训练，以便学习语言的统计规律。然而，将整个文本语料库一次性输入模型在计算上是不可行的。因此，训练是在小的、随机采样的文本块上进行的。本节解释了为什么我们要使用块、单个块如何已经包含多个训练样本，以及这些块如何被组织成批次以进行高效的 GPU 处理。

采样块与 block size

在训练期间，我们从数据集中随机采样小块文本，而不是使用完整文本。每个块都有一个最大长度，通常称为 block size（也称为上下文长度）。如果我们设置 block_size = 8，并查看训练数据的前九个字符，我们会得到如下序列：

# First block_size + 1 characters from the training set
# e.g., [18, 47, 56, 57, 58, 1, 15, 47, 58]

一个包含九个字符的块实际上内部打包了八个独立的训练样本，因为每个字符都跟随在它之前的字符后面。模型在每个位置同时被训练来预测下一个 token。例如，给定上下文 18，目标是 47；给定上下文 18, 47，目标是 56；依此类推，一直到八个 token 的上下文预测第九个。 [00:15:01]

以下是代码中提取这八个样本的方式：

x = train_data[:block_size]    # first block_size tokens
y = train_data[1:block_size+1] # offset by one

for t in range(block_size):
    context = x[:t+1]
    target = y[t]
    print(f"when input is {context.tolist()} the target: {target}")

运行这段代码会打印出隐藏在这个单独九字符块中的八个独立样本。

为什么要在所有上下文长度上训练

我们刻意在长度从 1 一直到完整 block size 的上下文上进行训练。这不仅仅是为了效率——它是一个刻意的设计选择。通过让 Transformer 接触到 block 内每一种可能的上下文长度，我们确保它学会在给定一个 token 的历史或最大允许长度时都能做出预测。 [00:17:08] 这在推理阶段变得至关重要：生成可以仅从单个字符的上下文开始，而模型已经知道如何继续。一旦运行中的上下文超过 block_size，旧的 token 就会被截断，因为模型在预测下一个字符时永远不会接收超过 block_size 个输入。

添加 batch 维度

最后一个要素是 batch 维度。虽然时间维度（block_size）定义了模型在序列中看到多少个 token，但 batch 维度将多个独立的块堆叠到一个张量(tensor)中。这样做纯粹是为了计算效率：GPU 擅长并行处理，因此同时处理多个块能让它们保持忙碌。这些块是完全独立的，彼此之间没有交互。 [00:18:11]

以下函数构建了一个批量的数据加载器。它首先设置一个随机种子以保证可复现性，然后在数据集中采样 batch_size 个随机偏移量。从每个偏移量处，它抓取 block_size 个 token 作为输入 x，以及接下来的 block_size 个 token（偏移一位）作为目标 y，并将它们堆叠成一个 (batch_size, block_size) 的张量(tensor)：

torch.manual_seed(1337)
batch_size = 4
block_size = 8

def get_batch(split):
    data = train_data if split == 'train' else val_data
    ix = torch.randint(len(data) - block_size, (batch_size,))
    x = torch.stack([data[i:i+block_size] for i in ix])
    y = torch.stack([data[i+1:i+1+block_size] for i in ix])
    return x, y

xb, yb = get_batch('train')
print('inputs:')
print(xb.shape)  # torch.Size([4, 8])
print(xb)
print('targets:')
print(yb.shape)  # torch.Size([4, 8])
print(yb)

得到的 xb 是一个 4 × 8 的张量(tensor)：四行独立的行，每一行都是训练集的一个块。对应的 yb 保存了 xb 中每个位置的目标。 [00:19:44]

总的来说，这一个批次包含了 4 × 8 = 32 个独立的样本。例如，当输入包含 24 时，目标是 43；当输入是 24, 43 时，目标是 58；当输入是 24, 43, 58 时，目标是 5，依此类推，遍及所有行。这 32 个样本被打包到一个批次中，以便 Transformer 可以并行处理它们，尽管就模型而言它们仍然是完全独立的。 [00:20:48]

实现 bigram 语言模型

准备好一批 token 序列后，下一步就是引入最简单的神经语言模型：bigram 模型。在这种架构中，仅使用单个 token 的身份来预测其后跟随的内容。模型不考虑相邻 token 的上下文；每个 token 只查看自身，以估计序列中的下一个字符。

实现是一个极简的 PyTorch 模块。在构造函数内部，使用 torch.nn.Embedding 创建一个形状为 (vocab_size, vocab_size) 的 token 嵌入表(embedding table)。该表充当一个查找矩阵：每个整数 token 索引映射到一行 logits，代表对整个词汇表的分数。

import torch
import torch.nn as nn
from torch.nn import functional as F

torch.manual_seed(1337)

class BigramLanguageModel(nn.Module):
    def __init__(self, vocab_size):
        super().__init__()
        self.token_embedding_table = nn.Embedding(vocab_size, vocab_size)

    def forward(self, idx, targets=None):
        logits = self.token_embedding_table(idx)  # (B, T, C)
        return logits

m = BigramLanguageModel(vocab_size)
logits = m(xb, yb)
print(logits.shape)

当输入 idx 的形状为 (B, T) —— batch size 为 4，time steps 为 8 —— 经过嵌入表(embedding table)时，PyTorch 为每个 token 收集对应的行，并将结果排列成一个形状为 (B, T, C) 的张量(tensor)，其中 C 是 vocab_size（本例中为 65）。[00:22:53] 输出中的每个位置都被解释为下一个 token 的 logits，即分数。由于模型无法利用周围的上下文，它唯一的信号就是训练数据中字符彼此跟随的频率。

评估损失

为了衡量预测质量，负对数似然损失 —— 在 PyTorch 中以 F.cross_entropy 提供 —— 是自然的选择。直观地说，对应正确下一个 token 的 logit 应该很大，而所有其他的应该很小。[00:25:00]

PyTorch 期望的输入布局带来了一个复杂情况：对于多维输入，cross_entropy 要求 channel 维度排在第二位，即形状为 (B, C, T) 而不是 (B, T, C)。与其置换维度，不如将 logits 和 targets 分别重塑为二维和一维张量(tensor)来得简单：

B, T, C = logits.shape
logits = logits.view(B * T, C)
targets = targets.view(B * T)
loss = F.cross_entropy(logits, targets)
print(loss)

将 batch 和 time 维度拉伸成一个单一的序列维度后，形状就符合 F.cross_entropy 的要求了。[00:27:07] 运行代码得到的初始损失约为 4.74。

解读初始损失

对于一个包含 65 个字符的词汇表，一个完全均匀的随机模型会给每个 token 分配概率 $1/65$。在这种情况下，期望的交叉熵损失为

$$-\log \frac{1}{65} \approx 4.17.$$

观察到的损失 4.74 略高于这个理论下限。[00:28:10] 这表明随机初始化的嵌入表(embedding table)尚未学到数据中存在的任何 bigram 统计信息，也尚未进行优化。初始损失与随机猜测基线之间的差距证实了模型一旦开始训练就有改进的空间。

从模型生成

generate 方法接收一批形状为 (B, T) 的上下文索引(context indices)，并将每个序列沿时间维度扩展到 max_new_tokens。

def generate(self, idx, max_new_tokens):
    for _ in range(max_new_tokens):
        logits, loss = self(idx)
        logits = logits[:, -1, :]          # (B, C) — last time step only
        probs = F.softmax(logits, dim=-1)  # (B, C)
        idx_next = torch.multinomial(probs, num_samples=1)  # (B, 1)
        idx = torch.cat((idx, idx_next), dim=1)             # (B, T+1)
    return idx

在循环内部，模型返回每个时间步(time step)的 logits，但只有最后一步用于预测。这些最后一步的 logits 通过 softmax 转换为概率(probabilities)，然后 torch.multinomial 为每个批次元素采样一个 token。采样得到的索引被拼接到正在运行的上下文(context)上，并在下一次迭代中重新输入模型。 [00:29:43]

由于 generate 调用 forward 时不传入目标(targets)，forward 的签名必须将 targets 设为可选。当 targets 为 None 时，不计算损失(loss)，函数仅返回 logits。

为了启动生成，将一个保存整数 0（换行符 token）的 (1, 1) 张量(tensor)传入 generate。在生成 100 个新 token 后，去掉单一批次维度，将结果转换为 Python 列表，再由 decode 将索引映射回字符。 [00:32:26]

由于模型尚未训练，输出最初是随机的乱码。值得注意的是，实现中每一步都将整个不断增长的上下文输入模型，即使 bigram 模型只需要前一个 token。这种设计是有意的前瞻性考虑：一旦模型开始使用更长的上下文，同样的 generate 流程可以原样工作，无需改动。 [00:34:05]

通信问题

考虑一个形状为 [B, T, C] 的序列批次，其中 B 是 batch size，T 是时间维度（在这个 toy example 中最多八个 token），C 是通道数（这里只有两个）。[00:43:01] 每个 token 目前都是独立存在的；一个序列中的八个 token 互不交流。我们希望把它们耦合起来，但要遵守一条严格的规则：位置五的 token 只能从位置一到五接收信息。它必须对位置六、七、八保持“盲目”，因为那些是未来 token，而模型正要预测接下来会出现什么。[00:43:31]

最简单的方案：滑动平均

让 token 与其过去通信的最简单方法，就是对每个前面时间步的通道向量（包括它自己）取平均。如果你是第五个 token，你就收集步骤一到五的二维通道向量并求均值。这会得到一个新的特征向量，把你放在你的历史上下文中进行概括。

这是一种极其有损的交互形式——空间排列和个体身份都被抹去了——但它足以说明因果约束，而且我们后面会找回那些丢失的信息。[00:44:03]

要理解为什么这样做有效，先从一个普通的全1矩阵和一个随机数据矩阵开始。

import torch

a = torch.ones(3, 3)
b = torch.randint(0, 10, (3, 2)).float()
c = a @ b

因为 a 的每一行都是全1，c 的每个元素就只是 b 某一列的和。左上角元素是 2 + 6 + 6 = 14，旁边的是 7 + 4 + 5 = 16，依此类推。c 的每一行都重复相同的和，因为 a 的每一行都相同。

现在用 torch.tril 把 a 变成下三角矩阵。

a = torch.tril(torch.ones(3, 3))
c = a @ b

a 的第一行是 [1, 0, 0]，所以 c 的第一行就是 b 的第一行。a 的第二行是 [1, 1, 0]，所以 c 的第二行变成 b 前两行的和：2 + 6 = 8 和 7 + 4 = 11。a 的第三行是 [1, 1, 1]，所以 c 的第三行是 b 三行的总和。通过控制1出现的位置，每一输出行累积了可变数量的输入行。

到目前为止，这个操作产生的是和，而不是平均值。要把和变成平均值，把 a 的每一行归一化，使其元素和为1。

a = torch.tril(torch.ones(3, 3))
a = a / torch.sum(a, dim=1, keepdim=True)
c = a @ b

归一化后，第一行是 [1]，第二行是 [0.5, 0.5, 0]，第三行是 [1/3, 1/3, 1/3]。相乘后得到：第一行是 b 的第一行不变，第二输出行是前两行的平均，第三输出行是三行的平均。循环所做的那种增量式因果平均，现在被表达为单个矩阵乘积。

把这个方法应用到实际的 token 批次上，权重变成一个 T × T 的下三角矩阵，其每一行和为1，数据矩阵是 token 嵌入(embedding) x，形状为 (B, T, C)。PyTorch 自动广播批次维度，并执行批处理矩阵乘法。

wei = wei / wei.sum(dim=1, keepdim=True)
xbow2 = wei @ x   # (B, T, T) @ (B, T, C) -> (B, T, C)

因为形状不同，PyTorch 会为 wei 插入一个批次维度，在所有批次元素上独立应用乘法，并输出形状为 (B, T, C) 的结果。结果与慢速循环版本完全相同，但对每个批次元素和每个 token 位置都并行计算。 [00:47:10] [00:48:48] [00:50:23] [00:51:59] [00:53:32]

Softmax 形式化表述

首先，将权重矩阵全部初始化为零：

然后，我们用负无穷大掩掉上三角（未来部分）：

这里的 tril 是值为 1 的下三角掩码。tril 为零的每一个位置——也就是每一个来自未来的 token——都变成 -inf。 [00:55:05]

最后，我们在最后一个维度上应用 softmax：

因为 $\exp(-\infty)=0$，被掩码的位置在指数化后不再贡献任何值。剩下的位置原本都是 $0$，所以 $\exp(0)=1$。逐行归一化后，就得到了与之前完全相同的三角权重模式：第一行是 $[1, 0, 0, \dots]$，第二行是 $[0.5, 0.5, 0, \dots]$，依此类推。 [00:56:06]

为什么这种表述方式很重要

从这个角度看，wei 中的每一项都是亲和度(affinity)——交互强度，表示每个 token 希望从其他每个 token 聚合多少信息。 [00:56:37] masked_fill 步骤强制执行因果约束：未来 token 被钳制到 $-\infty$，因此它们的亲和度被强制为零，无法与过去通信。Softmax 对剩余部分进行归一化，将原始亲和度转化为对过去位置的合法概率分布，随后的矩阵乘法则执行加权聚合。

预览：数据相关的亲和度

在当前版本中，每一个非对角线的亲和度都只是零——一个由我们选定的常数。关键的预告是，在自注意力(self-attention)中，这些零将变成数据相关的。Token 会互相审视，并基于它们自身的值来决定彼此之间的感兴趣程度。某些配对会获得高亲和度，另一些则较低，而这一决定将从数据中学习，而非硬编码。 [00:57:09]

一旦这些亲和度被设定，同样的流程仍然适用：掩码未来、用 softmax 归一化、聚合。唯一的变化是权重不再是均匀且固定的；它们由模型自身动态分配。

从均匀平均到数据相关的亲和性

我们现在来到了自注意力(self-attention)的核心：实现单个自注意力头(self-attention head)。设定是一个包含四个序列的批次(batch)，每个序列有八个词元(token)，每个词元都是 32 维的向量。到目前为止，代码仅通过简单地对所有过去和当前词元进行均匀平均来混合信息。这是通过构造一个下三角掩码(lower-triangular mask)，将被掩码的位置填充为负无穷，然后应用 softmax 来实现的。当亲和矩阵(affinity matrix)初始化为零时，所得权重的每一行都会变得均匀，这意味着每个词元从每个先前的词元及其自身收集的信息量是相同的。 [01:02:24]

均匀平均是不够的，因为不同的词元需要以不同的程度发现其他词元更相关或更不重要。这种交互应该是数据相关的(data-dependent)：例如，一个元音词元可能需要特别关注前面的辅音，并让它们的信息流入自身。自注意力(self-attention)正是解决了这个问题。 [01:03:27]

其机制如下。每个位置上的每个词元都会发出两个学习得到的向量：一个query（查询）和一个key（键）。query 代表“我在寻找什么？”，key 代表“我包含什么？”。为了决定一个词元应该对另一个词元给予多少关注，我们计算当前词元的 query 与所有其他词元的 key 之间的点积(dot product)。 [01:03:58]

# x: (B, T, C)
# query and key projections
q = x @ W_q  # (B, T, head_size)
k = x @ W_k  # (B, T, head_size)

# affinities: dot product of queries with keys
wei = q @ k.transpose(-2, -1)  # (B, T, T)

如果一个 query 和一个 key 对齐得很好，它们的点积会很大且为正，从而产生一个高亲和权重(affinity weight)。该权重随后控制从对应词元收集多少信息。相反，不匹配的 query 和 key 会产生较小或负的点积，在经过掩码和 softmax 后变成接近零的权重，从而抑制不相关的词元。 [01:04:29]

这用学习得到的、内容相关的加权取代了均匀平均：每个词元现在根据数据本身选择要关注哪些过去的词元。

一个自注意力头将每个 token 转换为三个不同的向量：query、key 和 value。query 表达一个 token 在寻找什么，key 宣告一个 token 包含什么，而 value 则是当匹配成功时实际传递的信息。本节将带你完整走过单个头的正向传播过程，从投影到最终的加权聚合。

计算 token 到 token 的亲和度

为了衡量每个 token 应该关注其他 token 的程度，我们计算每个 query 与每个 key 之间的点积。张量(tensor)的形状为 (B, T, head_size)，因此我们在相乘前将 k 的最后两个维度转置：

结果是一个形状为 (B, T, T) 的方阵亲和度矩阵，对应 batch 中的每个序列。条目 wei[b, t, s] 衡量 token t 基于其 query 和 key 向量的相似性"关心" token s 的程度。与之前使用的均匀常数权重不同，这些亲和度是数据相关的：不同的 batch 元素、不同的 token 以及不同的位置都会产生不同的模式。[01:06:07]

检查单个样本（batch 元素 0），最后一行对应第八个 token。该 token 既知道自身内容也知道自身位置；它发出一个 query，例如"我是一个位于位置 E 的元音，正在寻找前四个位置中的辅音"。与此同时，前面的 token 发出 key。如果某个 key 匹配该描述，query 与 key 之间的点积就会变大，从而产生高亲和度。通过随后的 softmax，这种高亲和度使得第八个 token 拉取匹配 token 信息的大部分份额。[01:08:12]

因果掩码与 softmax 归一化

然而，位于位置 t 的 token 绝不能聚合来自位置 t+1 及之后的信息。我们通过一个上三角掩码来强制这一点，将未来位置设为负无穷，使它们在 softmax 后获得零权重。我们还对 logits 进行缩放，以保持梯度良好。[01:09:14]

经过掩码和指数化后，每一行之和为 1，得到一个干净的分布，以数据相关的方式精确告诉我们应该融入多少每个过去 token 的信息。[01:09:45]

聚合 value，而非原始 token

最后一个要素是 value 向量。每个 token 不是聚合原始输入 X，而是生成一个 value 投影，代表它愿意传达的内容：

v = value(X)   # (B, T, 16)
out = wei @ v  # (B, T, T) @ (B, T, 16) -> (B, T, 16)

你可以将 X 视为每个 token 持有的私有信息，而 v 是它广播的公开消息。如果另一个 token 觉得这个 token 很有趣（高 query-key 亲和度），它会收到 v 的加权副本，而不是 X 的。由于 v 存在于 16 维的头空间中，单个头的输出也是 16 维的。[01:10:17]

这完成了单头自注意力机制：投影到 key、query 和 value；计算缩放点积亲和度；掩码掉未来位置；用 softmax 归一化；并根据所得权重聚合 value。

概念深入：注意力作为通信机制，编码器 vs 解码器，自注意力 vs 交叉注意力

Attention 最好被理解为一种在有向图上的通信机制。想象一组节点，它们之间有边相连。每个节点持有一个信息向量，并通过加权求和的方式聚合所有指向它的节点中的数据。这些权重是数据依赖的：它们会根据任意时刻向量的实际内容动态变化。[01:11:50]

在我们自回归语言模型的设定中，图具有特定的三角结构。当块大小(block size)为八时，我们有八个 token，其连接遵循严格的规则：第一个节点只接收来自自身的边；第二个节点接收来自第一个节点和自身的边；依此类推，直到第八个节点，它接收来自所有前面节点以及自身的边。这种因果掩码(causal masking)确保每个位置只能关注前面的位置和自身。然而，attention 并不局限于这种布局。原则上，它可以应用于任意有向图，使其成为一种灵活的节点间通信原语。[01:12:20]

为什么位置编码(positional encodings)是必要的

Attention 的一个关键特性是，它作用于一个无序的向量集合。默认情况下，节点没有关于它们在序列中位于何处的概念。与卷积(convolution)不同——卷积的滤波器在固定的空间布局上操作，并通过其局部感受野隐式编码位置——attention 将其输入视为漂浮在空间中的一袋向量(bag of vectors)。如果你希望模型知道 token 3 在 token 5 之前，就必须显式注入位置信息，这正是我们向输入嵌入(embedding)添加位置编码(positional encodings)的原因。没有它们，模型将是置换不变的(permutation-invariant)，并且对词序一无所知。[01:12:51]

Batch 独立性

Batch 维度上的元素从不交互。尽管我们同时处理四个样本以提高效率，但批量化矩阵乘法(batch matrix multiplication)独立地对每个样本应用相同的操作。在有向图的类比中，batch size 为四意味着我们有四个独立的池，每个池包含八个节点。一个池内的八个节点可以自由通信，但没有任何边跨越池之间的边界。总共 32 个节点被并行处理，但它们仍然严格划分为四个独立的图。[01:13:54]

编码器块(encoder blocks) vs 解码器块(decoder blocks)

我们使用的三角掩码(triangular masking)是自回归语言建模特有的。它阻止未来的 token 向过去的 token 通信，否则会在训练期间泄露答案。但这个约束在一般情况下并非必需。对于情感分析等任务——其目标是从整个句子预测单个标签——让每个 token 关注所有其他 token 是完全合理且可取的。在这种情况下，你会使用 encoder block 的自注意力(self-attention)，它直接省略因果掩码：

# Encoder-style self-attention: all nodes talk to all nodes
wei = q @ k.T / sqrt(head_dim)
wei = F.softmax(wei, dim=-1)
out = wei @ v

我们一直在实现的是 decoder block。它通过应用三角掩码(triangular mask)来维持自回归结构，使得未来的节点永远不会与过去通信：

# Decoder-style self-attention: triangular mask preserves causality
wei = q @ k.T / sqrt(head_dim)
wei = wei.masked_fill(tril == 0, float('-inf'))
wei = F.softmax(wei, dim=-1)
out = wei @ v

两种设计都是有效的，底层的 attention 机制并不关心连接模式。它支持节点之间边的任意排列。[01:14:24]

自注意力(self-attention) vs 交叉注意力(cross-attention)

我们构建的操作具体来说是 self-attention，因为 keys、queries 和 values 都源自同一个源矩阵 x：

# Self-attention: Q, K, V all come from the same input x
q = x @ W_q
k = x @ W_k
v = x @ W_v

Attention 比这更通用。在 cross-attention 中，queries 仍然由当前输入产生，但 keys 和 values 来自一个独立的外部源。这种模式出现在编码器-解码器(encoder-decoder) Transformer 中，其中编码器处理输入序列（例如翻译中的源句子），解码器生成输出序列。解码器产生 queries，而 keys 和 values 取自编码器的最终表示。因此，解码器从旁边“读取”信息，而不是从自身读取：

# Cross-attention: queries from x, keys and values from an external context
q = x @ W_q
k = context @ W_k
v = context @ W_v

Self-attention 是外部源恰好与查询源相同的特例。Cross-attention 是从一组不同节点拉取信息的通用情况。[01:15:57]

按 sqrt(head_size) 缩放并将自注意力(self-attention)接入模型

自注意力(self-attention)机制已接近完成：query、key 和 value 已完成投影，分数(score)已计算，softmax 生成了聚合权重。按照 "Attention Is All You Need" 的表述，最后缺失的一块是缩放因子 $1 / \sqrt{d_k}$，其中 $d_k$ 即 head size。这种归一化并非仅仅为了美观；它控制了初始化时分数的统计行为，并防止 softmax 坍缩为 one-hot 分布。

为什么要按 $1 / \sqrt{\text{head_size}}$ 缩放？

假设输入到 key 和 query 投影的数据大致服从标准正态分布（零均值，单位方差）。因为每个注意力分数(score)都是一个点积 $q \cdot k$，该乘积的方差会随着向量维度的增加而增长。如果 head size 为 $d_k$，则原始分数的方差数量级约为 $d_k$ 本身。当 head size 为 16 时，分数的方差就是 16。乘以 $1 / \sqrt{d_k}$ 可以将方差重新缩放回 1，从而在初始化时保持预期的分布。

这一点很重要，因为分数会直接进入 softmax。当 softmax 的输入较小且集中在零附近时，输出是弥散的——每个位置获得适度的注意力份额。但如果输入的数值很大，softmax 就会急剧锐化到最大值，产生接近 one-hot 的向量。在这种情况下，每个节点只会从另一个节点聚合信息，从而丧失了注意力的意义。缩放使得初始化时的分数保持适中，确保模型在训练开始时拥有宽泛且有用的注意力模式，而不是尖锐、随意的模式。 [01:17:29]

实现 Head 模块

有了缩放的理论依据，Head 模块就可以实现为单个自注意力头(self-attention head)。它接受一个 head_size，并创建三个不带偏置的线性投影——key、query 和 value——这是标准做法。防止未来位置与过去位置通信的下三角掩码(lower-triangular mask)被注册为 PyTorch 缓冲区(buffer)而非参数(parameter)，因为它是固定的。

class Head(nn.Module):
    def __init__(self, head_size):
        super().__init__()
        self.key = nn.Linear(n_embd, head_size, bias=False)
        self.query = nn.Linear(n_embd, head_size, bias=False)
        self.value = nn.Linear(n_embd, head_size, bias=False)
        self.register_buffer('tril', torch.tril(torch.ones(block_size, block_size)))

    def forward(self, x):
        B, T, C = x.shape
        k = self.key(x)
        q = self.query(x)
        # Compute attention scores ("affinities")
        wei = q @ k.transpose(-2, -1) * C**-0.5  # scaled attention
        wei = wei.masked_fill(self.tril[:T, :T] == 0, float('-inf'))
        wei = F.softmax(wei, dim=-1)
        v = self.value(x)
        out = wei @ v
        return out

前向传播看起来应该很熟悉：投影到 key、query 和 value；计算缩放后的分数；应用因果掩码(causal mask)；softmax；然后聚合 value。关键的新增部分是乘以 C**-0.5，即 $1 / \sqrt{\text{head_size}}$，它实现了缩放点积注意力(scaled dot-product attention)。 [01:19:03]

将自注意力(self-attention)集成到语言模型中

要将注意力头(head)接入网络，只需在语言模型构造函数中实例化它，为简单起见，让 head size 等于嵌入维度(embedding dimension)。在 token 嵌入(token embedding)与位置嵌入(positional embedding)相加后，将结果送入自注意力头(self-attention head)，然后将输出传递给语言建模头(language modeling head)以生成 logits。

class BigramLanguageModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.token_embedding_table = nn.Embedding(vocab_size, n_embd)
        self.position_embedding_table = nn.Embedding(block_size, n_embd)
        self.sa_head = Head(n_embd)
        self.lm_head = nn.Linear(n_embd, vocab_size)

    def forward(self, idx, targets=None):
        B, T = idx.shape
        tok_emb = self.token_embedding_table(idx)
        pos_emb = self.position_embedding_table(torch.arange(T, device=device))
        x = tok_emb + pos_emb
        x = self.sa_head(x)
        logits = self.lm_head(x)
        ...

这是最简单的集成方式：一个单独的自注意力(self-attention)组件位于输入嵌入(input embeddings)和输出投影(output projection)之间。 [01:20:03]

在生成过程中遵守 block_size

generate 方法还需要一处额外修改。由于模型现在使用了位置嵌入(positional embeddings)，输入序列 idx 的长度绝不能超过 block_size。否则，位置嵌入表(position embedding table)会越界，因为它只存储了位置 $0$ 到 $\text{block_size}-1$ 的嵌入。为了防止这种情况，在将上下文输入模型之前，将其裁剪为最后 block_size 个 token。

def generate(self, idx, max_new_tokens):
    for _ in range(max_new_tokens):
        idx_cond = idx[:, -block_size:]  # crop to block_size
        logits, loss = self(idx_cond)
        logits = logits[:, -1, :]
        probs = F.softmax(logits, dim=-1)
        idx_next = torch.multinomial(probs, num_samples=1)
        idx = torch.cat((idx, idx_next), dim=1)
    return idx

有了这些组件——缩放后的注意力分数、完整的 Head 模块、将其插入语言模型，以及在生成时裁剪上下文——网络就准备好使用自注意力(self-attention)进行训练了。 [01:21:07]

Multi-Headed Self-Attention（多头自注意力）

A single self-attention head gives tokens one communication channel, but language is rich enough that different tokens need to discover many different relationships simultaneously. Multi-head attention（多头注意力）solves this by running several self-attention operations in parallel and concatenating their outputs, analogous to group convolutions in computer vision.

From One Channel to Many（从单通道到多通道）

Instead of one attention head that projects the $n_{\text{embed}}$-dimensional space into query, key, and value vectors, multi-head attention creates $h$ independent heads. Each head performs scaled dot-product attention on a lower-dimensional subspace. If $n_{\text{embed}} = 32$ and we use $h = 4$ heads, each head operates with a head size of $8$:

class MultiHeadAttention(nn.Module):
    def __init__(self, num_heads, head_size, n_embed):
        super().__init__()
        self.heads = nn.ModuleList([
            Head(head_size, n_embed) for _ in range(num_heads)
        ])

    def forward(self, x):
        # Run all heads in parallel and concatenate over the channel dimension
        out = torch.cat([h(x) for h in self.heads], dim=-1)
        return out

Each Head is the masked scaled dot-product attention from the previous section. The concatenation happens along the final (channel) dimension: four heads each output an 8-dimensional vector per token, which concatenates back to the original 32 dimensions. [01:22:09]

Why Multiple Heads Help（多头为何有效）

A single 32-dimensional attention must simultaneously learn to track consonants, vowels, positional patterns, and syntactic roles through one set of projections. By splitting into four 8-dimensional heads, the network creates independent pathways. One head might specialize in local vowel agreement, another in consonant clusters, a third in long-range positional dependencies. [01:23:41]

This is structurally similar to group convolutions: rather than one large filter bank operating on the full feature depth, grouped convolution partitions channels into independent groups that learn distinct filters. Multi-head attention applies the same principle to the communication pathway between tokens. [01:23:10]

Position-Wise 前馈计算

前馈子层是一个微小的单隐藏层 MLP，独立地作用于每一个 token。在代码中，它只是一个线性投影后接 ReLU 非线性激活：

class FeedFoward(nn.Module):
    """a simple linear layer followed by a non-linearity"""

    def __init__(self, n_embd):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(n_embd, n_embd),
            nn.ReLU(),
        )

    def forward(self, x):
        return self.net(x)

由于 nn.Linear 作用于最后一个维度，而 batch/time 维度保持不变，因此每个 token 位置都独立地运行这个 MLP。自注意力(self-attention)层是通信阶段：token 彼此查看并聚合上下文。前馈层是计算阶段：每个 token 独自反思刚刚收集到的信息。 [01:25:16]

我们在前向传播中把它直接插在 attention 之后：

x = self.sa_heads(x)  # communication
x = self.ffwd(x)      # computation

加入这一层后，训练得到的验证损失从 2.28 降到 2.24。生成的文本仍然粗糙，但趋势证实，增加逐 token 计算有助于模型利用经过 attention 后的上下文。 [01:26:18]

打包成 Transformer Block

下一步是将通信层和计算层组合成一个单一的可复用单元。一个 Block 接受嵌入(embedding)维度和所需的 head 数量，然后自动推导出 head 大小，使得 n_embd = n_head * head_size。对于我们的 32 维嵌入，使用 4 个大小为 8 的 head。 [01:27:21]

class Block(nn.Module):
    """Transformer block: communication followed by computation"""

    def __init__(self, n_embd, n_head):
        super().__init__()
        head_size = n_embd // n_head
        self.sa = MultiHeadAttention(n_head, head_size)
        self.ffwd = FeedFoward(n_embd)

    def forward(self, x):
        x = self.sa(x)
        x = self.ffwd(x)
        return x

定义好 block 后，主模型就变成了这些 block 的顺序堆叠，最后接上解码器：

self.blocks = nn.Sequential(
    Block(n_embd, n_head=4),
    Block(n_embd, n_head=4),
    Block(n_embd, n_head=4),
    nn.Linear(n_embd, vocab_size),
)

这与 Transformer 架构一致：通信（多头自注意力）和计算（position-wise 前馈）交替重复。 [01:26:49]

然而，简单地堆叠很多 block 并不能立刻带来好结果。随着网络加深，它开始遇到优化困难。解决这些问题还需要原始 Transformer 论文中的另一个思路，我们将在下一章介绍。 [01:28:23]

残差（跳跃）连接

随着层数不断堆叠，深层网络变得越来越难以优化。梯度必须反向传播穿过每一次矩阵乘法、非线性和归一化，而在每一步都可能消失或爆炸。两项架构创新让深层 transformer 保持可训练性；第一项是残差（或跳跃）连接，出自 2015 年的论文 "Deep Residual Learning for Image Recognition"。[01:28:23]

残差路径

不要把网络想象成垂直堆叠的层，而是一条从输入直通输出的主干道。在任意位置，你都可以分叉出去执行一些变换，然后通过相加合并回主干道。原始输入几乎原封不动地一路传到最终预测，仅由一连串加法操作承载，而各个学习到的模块沿途贡献微小的修正。[01:28:53]

这种设计之所以强大，是因为加法在反向传播中的行为。在我们之前构建的 micro-grad 实现中，加法节点会将传入的梯度均等地分发给两个输入。因此，来自损失的梯度可以跳过每一个加法节点，直接流向最早的层，不会被侧分支上的权重矩阵阻碍。侧分支仍然能收到梯度，但它们不会阻塞主通道。[01:29:25]

在初始化时，侧分支通常被配置为对残差路径贡献极小——有时几乎为零。网络在诞生之初几乎是一个恒等映射，只有在优化过程中，残差块才会"上线"并开始贡献有意义的变换。由于梯度从第一天起就能到达最早的层，即使网络深达数十层甚至数百层，仍然保持可优化性。[01:29:58]

实现残差连接

在 transformer 块内部，我们将子层的输出加回到它们的输入上。对于多头注意力(multi-head attention)和前馈网络(feed-forward network)，模式相同：从残差路径分叉，计算，然后通过相加投影回去。

class Block(nn.Module):
    """One transformer block with residual connections around attention and feed-forward."""

    def __init__(self, n_embd, n_head):
        super().__init__()
        head_size = n_embd // n_head
        self.sa = MultiHeadAttention(n_head, head_size)
        # projection back into the residual pathway after attention
        self.proj = nn.Linear(n_embd, n_embd)
        self.ffwd = nn.Sequential(
            nn.Linear(n_embd, 4 * n_embd),
            nn.ReLU(),
            # projection back into the residual pathway after feed-forward
            nn.Linear(4 * n_embd, n_embd),
        )

    def forward(self, x):
        # residual connection around attention
        x = x + self.proj(self.sa(x))
        # residual connection around feed-forward
        x = x + self.ffwd(x)
        return x

注意力头(attention heads)产生大小为 n_embd 的拼接输出，我们先通过线性投影 self.proj，再将其加回残差流(residual stream)。前馈层(feed-forward layer)将维度扩展四倍，应用 ReLU，然后再投影回 n_embd，以保证相加时的形状兼容。[01:30:29]

这种四倍扩展与原始 transformer 论文一致，其中输入和输出维度为 512，内部前馈层为 2048。在残差分支内部增大中间表示可以增加计算量，同时不干扰主梯度 highway。[01:31:32]

有了这些模块，网络成功训练，验证损失下降到 2.08。由于模型现在足够大，训练损失开始超过验证损失——这是容量不再是瓶颈、正则化(regularization)即将变得重要的早期信号。[01:32:02]

Layer Normalization（层归一化）

训练极深网络的第二个关键创新是层归一化(layer normalization)。残差连接(residual connections)为梯度提供了一条高速公路，而层归一化则稳定了激活值本身，使其尺度在整个网络中保持受控。[01:32:33]

From Batch Norm to Layer Norm（从批归一化到层归一化）

层归一化(layer normalization)与批归一化(batch normalization)密切相关，后者我们在 makemore 系列中已经实现过。批归一化确保在批(batch)维度上，每一个单独的神经元都具有单位高斯分布：均值为零、标准差为一。换句话说，它对激活矩阵的列(columns)进行归一化。

回顾 makemore 中的 BatchNorm1d 模块。给定一个包含 32 个样本的批(batch)，每个样本是一个 100 维向量，批归一化保证第 0 列（以及所有其他列）是零均值、单位方差的：

# BatchNorm normalizes COLUMNS of x (across the batch dimension)
class BatchNorm1d:
    def __init__(self, dim):
        self.gamma = torch.ones(dim)
        self.beta = torch.zeros(dim)
        # ... running buffers for train/test distinction ...

    def __call__(self, x):
        # x is (Batch, Time, Channels) or (Batch, Channels)
        xmean = x.mean(0, keepdim=True)  # average over batch
        xvar = x.var(0, keepdim=True)
        xhat = (x - xmean) / torch.sqrt(xvar + 1e-5)
        self.out = self.gamma * xhat + self.beta
        return self.out

默认情况下，行(rows)不会被归一化，因为均值和方差是沿批(batch)维度（维度 0）计算的。[01:33:04]

Implementing Layer Norm（实现层归一化）

层归一化(layer normalization)将这一点翻转过来：不再归一化列(columns)，而是归一化行(rows)。对于每一个单独的样本，其特征向量被归一化为零均值、单位方差。实现方式几乎完全相同——只有归一化的维度发生了变化：

class LayerNorm1d:
    def __init__(self, dim):
        self.gamma = torch.ones(dim)
        self.beta = torch.zeros(dim)

    def __call__(self, x):
        # x is (Batch, Time, Channels) or (Batch, Channels)
        xmean = x.mean(1, keepdim=True)  # average over features (rows)
        xvar = x.var(1, keepdim=True)
        xhat = (x - xmean) / torch.sqrt(xvar + 1e-5)
        self.out = self.gamma * xhat + self.beta
        return self.out

由于计算不再跨样本进行，我们可以删除所有运行缓冲区(running buffers)。训练时和测试时没有区别，没有动量(momentum)，也不需要跟踪运行统计量(running statistics)。只剩下可学习的 gamma 和 beta 参数。[01:34:06]

Pre-Norm vs. Post-Norm（预归一化与后归一化）

原始的 Transformer 论文将 "Add & Norm" 放在子层变换之后（post-norm）。然而，现代实践已经转向预归一化(pre-norm)形式，即层归一化(layer normalization)在变换之前应用。这种重新排序已成为标准，也是我们这里实现的方式。[01:35:08]

在 Block 中，我们添加两个 LayerNorm 模块，并在自注意力(self-attention)之前和前馈网络(feedforward network)之前将它们应用于 x：

class Block(nn.Module):
    def __init__(self, n_embd, n_head):
        super().__init__()
        head_size = n_embd // n_head
        self.sa = MultiHeadAttention(n_head, head_size)
        self.ffwd = FeedForward(n_embd)
        self.ln1 = nn.LayerNorm(n_embd)
        self.ln2 = nn.LayerNorm(n_embd)

    def forward(self, x):
        x = x + self.sa(self.ln1(x))   # pre-norm before attention
        x = x + self.ffwd(self.ln2(x)) # pre-norm before feedforward
        return x

层归一化(layer norm)的大小是 n_embd（例如 32）。均值和方差在这 32 个特征值上计算，因此批(batch)维度和时间(time)维度都充当批维度。这是一种逐 token 的变换，将特征归一化并在初始化时使它们成为单位高斯分布。层归一化内部可学习的 gamma 和 beta 允许优化器在训练期间按需缩放和平移归一化后的值。[01:36:13]

Final Decoder Layer Norm（最终的解码器层归一化）

通常还在 Transformer 的最末端、最终线性层之前再放置一个层归一化(layer norm)，该线性层将结果解码为词表 logits：

self.blocks = nn.Sequential(*[Block(n_embd, n_head) for _ in range(n_layer)])
self.ln_f = nn.LayerNorm(n_embd)  # final layer norm
self.lm_head = nn.Linear(n_embd, vocab_size)

加上这一层，decoder-only Transformer 的主要组件就全部就位了。使用层归一化(layer normalization)训练后，验证损失(validation loss)达到 2.06，比之前的 2.08 略有提升。在更大、更深的网络中，收益会更加明显。[01:37:17]

Dropout 作为正则化(regularization)技术

Dropout 在每个 block 内部的三个位置应用：在从 feed-forward 分支返回的残差加法(residual addition)之前、在多头注意力(multi-head attention)计算结束时，以及在产生注意力亲和度(attention affinities)的 softmax 之后立即应用。通过随机将部分 post-softmax 权重置零，网络临时阻止某些节点相互通信。[01:38:17]

该机制在 2014 年的一篇论文中提出，工作原理如下：在每次前向-后向传播(forward-backward pass)中，随机掩码(random mask)通过将固定比例的神经元置零来禁用它们。由于掩码在每次传递时都会变化，训练过程实际上优化了许多较小子网络的集成(ensemble)。在推理(inference)时使用完整网络，这相当于将该集成合并为单个模型。[01:38:49] 对于当前目的，将 dropout 视为一种正则化器(regularizer)就足够了，它在模型增长到足以过拟合(overfit)时变得重要。

将我们的 decoder-only 模型映射到完整的 encoder-decoder "Attention Is All You Need" 架构

如果你回顾原始的 "Attention Is All You Need" 图示，会注意到一些在我们的实现中从未出现的额外部分：左侧的 "encoder" 堆栈以及每个 decoder 块内部的 "cross-attention" 子层。这些部分确实存在，但我们的模型有意设计得更简单。我们构建的是一个 decoder-only Transformer——与 GPT 属于同一家族——而原始论文展示的是一个为机器翻译设计的 encoder-decoder Transformer。

The full encoder-decoder Transformer architecture from 'Attention Is All You Need', annotated with a French-to-English translation example showing how the encoder processes the source sentence and the decoder generates the target sentence. [01:43:00]

为什么我们的 block 看起来不同

我们的 Block 只包含两个子层：self-attention 和 feed-forward network。原始的 decoder block 包含第三个子层 cross-attention，夹在它们之间。我们省略了它，因为我们的任务是无条件文本生成：我们输入一串 token 序列，让模型预测下一个 token，没有需要关注的外部信息源。 [01:42:32]

使一个堆栈成为 decoder 的是 三角（因果）掩码。在 self-attention 中，我们屏蔽掉所有未来位置，使得位置 $i$ 只能关注位置 $\leq i$。这种自回归特性让我们可以从模型中逐个 token 采样：

# causal mask used in our decoder-only model
# tril creates a lower-triangular matrix of ones
wei = wei.masked_fill(tril == 0, float('-inf'))  # [T, T]
wei = F.softmax(wei, dim=-1)

因为每个 token 只关注自身和之前的 token，模型可以通过简单的 next-token prediction 进行训练，并通过从最终的 softmax 采样来生成文本。 [01:43:03]

为什么原始论文有一个 encoder

原始的 Transformer 是为 机器翻译 构建的。在这种设定下，模型必须将源语言句子（例如法语）翻译成目标语言句子（例如英语）。源语言句子不是生成的；它在开始时就完整给出。 [01:43:34]

一个典型的训练示例如下：

  • Encoder input：一串法语 token，包括标记句子开头和结尾的特殊 token。
  • Decoder input：一个特殊的 <start> token，后面跟着到目前为止生成的英语 token。
  • Target output：英语句子，以一个特殊的 <end> token 结尾。

Decoder 生成 "Neural networks are awesome" 的方式与我们的模型生成文本的方式完全相同，但有一个关键区别：生成过程 以 法语源句 为条件。 [01:44:05]

Encoder 和 cross-attention 如何工作

Encoder 读取整个法语句子，对其进行 tokenize，并将其通过它自己的 Transformer 堆栈。Encoder 中没有三角掩码，因此每个源 token 都可以关注其他每个源 token。Encoder 的工作仅仅是构建源语义丰富、双向的表示。 [01:44:36]

一旦 encoder 完成，它的最终输出向量会被传递给 decoder。在每个 decoder block 内部，cross-attention 层计算 attention，其中：

  • Queries 来自 decoder 自身的隐藏状态（部分生成的英语句子）。
  • Keys 和 Values 来自 encoder 的输出向量（编码后的法语句子）。

从数学上讲，如果 $X$ 是 decoder 状态，$E$ 是 encoder 输出：

$$\text{CrossAttention}(X, E) = \text{softmax}\left(\frac{X W_q \,(E W_k)^T}{\sqrt{d_k}}\right) E V_v$$

来自 encoder 的 keys 和 values 被送入 每一个 decoder block，因此整个解码过程始终锚定在源句子上。 [01:45:07]

从 1000 万参数模型到 GPT-3 的 1750 亿参数

Shakespeare Transformer 包含大约 1000 万参数，并在约 100 万字符上训练。由于 OpenAI 使用了一个包含约 50,000 个词元(token)的子词(sub-word)分词器(tokenizer)，同样的 Shakespeare 语料库会被压缩到约 300,000 个词元(token)。相比之下，GPT-3 的最大变体拥有 1750 亿参数，并在 3000 亿词元(token)上训练——数据量增加了约一百万倍，模型尺寸增加了 17,500 倍。 [01:49:17]

GPT-3 论文中的超参数应该看起来很熟悉。最大的模型使用 96 层、12,288 维的嵌入(embedding)维度、96 个注意力(attention)头以及 128 的头维度。学习率与 Shakespeare 模型使用的处于同一范围，架构与这里构建的实现几乎相同。差异主要是一个基础设施挑战：在这种规模下训练需要数千个 GPU 在分布式环境中通信。 [01:50:20]

预训练(pre-training)：在互联网规模数据上进行文档补全

创建 ChatGPT 这类系统的第一阶段是预训练(pre-training)。在此阶段，一个仅解码器(decoder-only)的 Transformer 在互联网的大规模数据切片上训练，以预测序列中的下一个词元(token)。结果是一个文档补全器(document completer)：它生成类似于训练分布的文本，但并未针对任何特定任务进行对齐。如果你向它输入一个问题，它可能会用另一个问题回应、继续一篇新闻文章，或生成任何在互联网上看起来合理的其他续写。 [01:51:22]

这是 nanoGPT 所关注的阶段。训练循环、检查点(checkpointing)、学习率衰减、模型编译以及跨多个节点和 GPU 的分布式训练都位于 train.py 中，而 model.py 包含与这里实现的相同的因果自注意力(causal self-attention)、MLP 和 Transformer 块。主要区别在于，nanoGPT 将所有注意力(attention)头批处理为一个额外的张量(tensor)维度，而不是循环遍历单个头，这在数学上是等价的但效率更高。 [01:46:42]

微调(fine-tuning)与对齐(alignment)：将基础模型转变为助手

预训练(pre-training)之后，必须对模型进行对齐(alignment)，使其表现得像一个有用的助手，而不是不受约束的文档补全器(document completer)。OpenAI 将这个对齐(alignment)阶段描述为三个宽泛的步骤。 [01:52:25]

监督微调(Supervised fine-tuning, SFT)。 第一步收集问答格式的演示数据。人类合同工编写提示(prompt)和期望的回复，创建一个比预训练(pre-training)语料库小得多的数据集——规模在数千到数万个示例的量级。然后在这个数据上对预训练(pre-trained)模型进行微调(fine-tuned)，使其学会在上下文(context)顶部期待一个问题，并在其下方生成答案。大型模型在这个阶段表现出惊人的样本效率。 [01:52:58]

奖励模型训练(Reward model training)。 第二步收集对比数据。对于给定的提示(prompt)，模型生成几个候选回复，人类评分员按质量对它们进行排序。这些排序被用来训练一个单独的奖励模型(reward model)，该模型对任何候选回复可能有多理想进行打分。 [01:53:29]

基于人类反馈的强化学习(Reinforcement learning from human feedback, RLHF)。 第三步使用奖励模型(reward model)通过 PPO（Proximal Policy Optimization，近端策略优化）对策略(policy)进行微调(fine-tune)，PPO 是一种策略梯度(policy-gradient)强化学习算法。采样策略(sampling policy)被更新，使得生成的答案预期能获得高奖励分数。这将模型从仅仅生成可能性高的文本推向人类评分员更偏好的文本。 [01:53:29]

这些对齐(alignment)阶段中使用的许多数据是 OpenAI 内部的，不公开可用，这使得微调(fine-tuning)阶段比预训练(pre-training)阶段更难复现。尽管如此，概念上的流水线是相同的：从一个文档补全器(document completer)开始，然后通过监督微调(supervised fine-tuning)、奖励模型(reward modeling)和强化学习(reinforcement learning)将其对齐(align)到特定行为。 [01:54:02]

任何超越纯语言建模的任务——情感分类、任务执行或助手式行为——都需要在基础模型之上进行额外的微调(fine-tuning)。Shakespeare 示例和 nanoGPT 只捕捉了这个流水线的第一阶段，但它们提供了构建其余部分所需的架构基础。 [01:55:35]
